Technical Report on Local-First Agentic Memory: Implementation and Optimization of the Query Markup Documents (qmd) FrameworkThe emergence of autonomous agentic systems has necessitated a transition from simple prompt-response cycles to complex, stateful interactions that require both durable memory and precise context retrieval. As technical ecosystems expand, the traditional approach of injecting entire repositories into a Large Language Model (LLM) context window has proven unsustainable, leading to significant token saturation and a phenomenon characterized as the working memory bottleneck. The research indicates that end-to-end multimodal models experience performance decay when subjected to excessive noise, particularly in long-context scenarios where critical information is buried in massive streams of data. Consequently, specialized local-first retrieval engines have become essential for maintaining agentic performance while ensuring data privacy and cost-efficiency. Query Markup Documents (qmd), a tool developed by Tobi Lütke, represents a paradigm shift in this domain by providing a local, high-accuracy search sidecar designed specifically for agentic flows.The Architecture of Query Markup DocumentsThe fundamental value proposition of qmd lies in its ability to transform an agent's interaction with a codebase from a passive "filing cabinet" model to an active "search engine" model. This is achieved through a multi-stage retrieval pipeline that integrates traditional keyword matching with advanced semantic reasoning. The qmd framework is built upon the premise that neither lexical nor semantic search is sufficient in isolation; lexical search (BM25) often fails to capture paraphrased concepts, while vector search frequently misses exact technical identifiers or error codes.The retrieval process within qmd follows a rigorous sequence: initial query expansion, parallel search via BM25 and vector embeddings, fusion using Reciprocal Rank Fusion (RRF), and a final LLM-based re-ranking stage. The system utilizes three local GGUF models, totaling approximately 2GB in size: an embedding model (e.g., embeddinggemma-300M), a cross-encoder re-ranker (e.g., Qwen3-Reranker-0.6B), and a query expansion model (e.g., qmd-query-expansion-1.7B). By running these models locally via node-llama-cpp, qmd avoids cloud API latencies and associated costs while maintaining strict data sovereignty.Pipeline StageMechanismObjectiveQuery ExpansionLocal LLMTransforms a single prompt into multiple intent-aware sub-queries.Lexical SearchSQLite FTS5 (BM25)Identifies exact matches, phrases, and handles term exclusions.Vector Searchsqlite-vecProvides semantic recall based on high-dimensional embeddings.FusionReciprocal Rank FusionMerges ranked lists from lexical and vector backends without parameter tuning.Re-rankingCross-Encoder LLMScores top candidates for final relevance to the specific user intent.The integration of RRF is a critical architectural choice, as it is a parameter-free method for combining different search signals. The research highlights that RRF handles missing signals gracefully, ensuring that if one backend fails to provide relevant results, the overall system remains robust. Furthermore, the recent updates to qmd have optimized this process by introducing adaptive parallelism, which computes the number of re-ranking contexts based on available VRAM or CPU cores, leading to a 2.7x performance increase on multi-core machines.Operational Syntax and Context ManagementThe operational efficacy of qmd is largely dependent on its structured query format. Conversely, Milvus represents the high-end of the spectrum, designed for distributed, multi-user environments that require horizontal scaling and billions of vectors.The Principles of Progressive DisclosureThe design of effective agentic memory systems is rooted in the concept of progressive disclosure. Originally a usability principle from the 1990s, it focuses on showing users—or in this case, agents—only the information they need at the immediate moment, deferring advanced details until they are requested. In the context of AI agents, this translates to a three-layer architecture designed to minimize token usage and maximize reasoning focus.The Index Layer: This layer contains lightweight metadata, such as titles, short descriptions, and capability tags. Claude Code's implementation of "Skills" uses this pattern by loading only skill names and descriptions at startup, keeping initialization fast and the base context lean.The Activation Layer: Full content is only loaded when a specific request matches a skill or documentation description. This creates an explicit control point, preventing the agent from being overwhelmed by irrelevant context.The Deep Dive Layer: Supporting materials, such as API documentation and reference scripts, are accessed only when the agent needs to perform execution or detailed troubleshooting. Tools like qmd are critical here, as they allow the agent to fetch exactly what is needed through a targeted search rather than reading entire directories.Implementing this principle requires a well-structured repository where instructions are decentralized.
