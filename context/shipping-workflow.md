# Tagging & Shipping Workflow

## App Repos (micro-services/*)

### Pre-tag checklist (mandatory before every tag)
1. Detect primary branch: `git symbolic-ref refs/remotes/origin/HEAD` ‚Üí extract name. Fallback: REPO_MAPPING `main_branch`, or convention (apps=`main`).
2. `git fetch origin`
3. `git rebase origin/{primary-branch}` ‚Äî abort and alert user on conflicts.
4. Get latest tag: `git tag --sort=-v:refname | head -1`
5. Only then create the new tag.

### Tag versioning
- Format: `x.y.z-dev` (dev/untested) or `x.y.z` (prod-ready).
- Default: always tag `-dev`. Increment patch from last tag (e.g., `0.0.4-dev` ‚Üí `0.0.5-dev`).
- Prod promotion: only when user explicitly says a `-dev` tag is tested and prod-ready. Drop `-dev` suffix, same patch (e.g., `0.0.5-dev` ‚Üí `0.0.5`). This is rare ‚Äî `-dev` tags in prod is normal and the default.
- No tags yet ‚Üí start at `0.0.1-dev`.

### Image build (JIB)
- **All Java services use JIB** (Maven plugin `jib-maven-plugin`). No Dockerfiles for deployment.
- Build and push command: `mvn compile jib:build -Djib.to.tags=X.Y.Z-dev -DskipTests`
- JIB pushes directly to Artifact Registry ‚Äî no manual `docker build`/`docker push`.
- The app tag and the image tag must match (e.g., git tag `0.0.6-dev` ‚Üí image `service-name:0.0.6-dev`).

## DAC/Terraform Repos (dac-sprvsr-core-*)
- **NEVER create git tags on DAC repos.** DAC repos do not use git tags.
- "Tag" in DAC context means the **CI/CD variable `TF_VAR_IMAGE_TAG`** in GitLab. This controls which Docker image version the Cloud Run service deploys.
- DAC deployment workflow: update `TF_VAR_IMAGE_TAG` in GitLab ‚Üí trigger pipeline on `dev`.
- **NEVER commit directly to `dev`, `uat`, or `main`.** Always create a branch named after the ticket (e.g., `SPV-4`), then create a merge request.
- **NEVER modify `README.md`** in DAC repos ‚Äî it is auto-generated by terraform-docs. Use `README_local.md` (gitignored) for local notes.

## Branch conventions
- DAC/Terraform repos (`dac-sprvsr-core-*`): primary = `dev`. Also have `uat` and `main` for promotion.
- App repos (`micro-services/*`): primary = `main`. Always verify via `origin/HEAD`.
- Branch naming: use ticket ID (e.g., `SPV-4`), not `feature/` prefix.

## PR Review with supervisr-github-review Agent

When using the `supervisr-github-review` subagent to review PRs:

### Draft-only ‚Äî NEVER post comments to GitHub
- The agent must NEVER post comments, reviews, or reactions to PRs
- Do NOT include instructions about posting in the agent prompt
- The agent drafts comments ‚Üí user reviews ‚Üí user posts manually via GitHub UI

### Comment format
Each draft comment must include:
1. **File and line(s)**: e.g., `SupervisorsDatafetcher.java:39`
2. **Comment text**: charitable, question-based (see phrasing rules below)
3. **GitHub suggestion block**: a fenced code block with the `suggestion` tag showing the proposed fix

### Phrasing rules ‚Äî natural variation required
- Vary sentence openers across comments. Do NOT start every comment the same way
- Use synonyms and varied structures: "I notice‚Ä¶", "Looking at this‚Ä¶", "Quick question on‚Ä¶", "This caught my eye‚Ä¶", "Curious about‚Ä¶", "Something worth checking‚Ä¶"
- Similarly vary closers: "Was this intentional or a miss?", "Is this the intended behavior?", "Worth a double-check?", "Thoughts?"
- The goal is to sound like a human reviewer, not a template

### Scope ‚Äî high-impact only
- Only flag bugs, logic errors, schema mismatches, security issues
- Skip style nits, formatting, naming preferences
- Assume good intent ‚Äî frame as questions, not accusations
